# Linux Subsystem

本内核是一个基于能力的混合内核, 因此, 内核几乎所有的系统调用都有一个主能力作为操纵的对象, 系统调用通过检查能力的类型和权限来决定是否允许操作, 以及如何操作. 这与 Linux 的系统调用模型有很大不同, 因此, 为了兼容 Linux 的应用程序, 内核提供了一个 Linux 子系统, 该子系统实现了 Linux 的系统调用接口, 并将其映射到内核的能力模型上.

具体而言, Linux Subsystem 是一个用户态程序. 为了实现 Linux 兼容, 内核特地只选取了至少以 0xFFFF 开头的系统调用号作为本系统的系统调用号, 以方便地分辨本内核的调用与 linux 系统调用. 当用户态程序发起一个 Linux 系统调用时, 如果这个程序是一个 Linux 兼容程序, 内核会将这个调用转发到 Linux Subsystem, 由 Linux Subsystem 来处理这个调用. Linux Subsystem 会根据系统调用号和参数, 调用内核的能力模型来完成操作, 并将结果返回给用户态程序.

Linux Subsystem 的原理是: 其本身实际上是一个可执行文件, 加载到一个特定的避免与用户程序冲突的地址中, 有着自己独立的堆, 但是与用户程序共用栈与内存空间, 实质上与 Linux 兼容程序是同一个进程, 只是有着相对独立的代码段和数据段. 其入口点同时承担两个功能: 1. 作为 Linux 兼容程序的入口点, 2. 作为 Linux Syscall Dispatcher 的入口点. 当用户程序发起一个 Linux 系统调用时, 内核会将调用转发到 Linux Subsystem 的入口点, 由其来处理这个调用. 由于 Linux Subsystem 与用户程序共用栈与内存空间, 因此, Linux Subsystem 可以直接访问用户程序的内存空间, 并且可以直接操作用户程序的栈, 以便于实现系统调用的参数传递和返回值传递.
系统将会将 t6 寄存器设置为 Linux Subsystem 的返回地址, 当 Linux Subsystem 处理完系统调用后, 会通过 jalr t6 指令返回到用户程序的下一条指令, 以便于继续执行用户程序. 可惜的是, 这么做一定会破坏 t6 寄存器的值, 因此 Linux Subsystem 并不是一个完美的解决方案. 但由于 t6 寄存器本身是编号相对靠后的临时寄存器, 在编译编译时不容易长时间保存值, 因此, 破坏 t6 寄存器的值基本上并不会对用户程序造成太大的影响. 要真正实现 Linux 兼容, 最好的办法是重写 musl 或者 glibc, 通过添加一个 linux 兼容层, 将其中涉及到 Linux 系统调用的部分, 改为调用兼容层内部的函数, 这些函数再调用内核的能力模型来完成操作. 这样就可以完全避免破坏 t6 寄存器的问题, 并且可以更好地实现 Linux 兼容. 但是, 一方面这需要大量的工作量, 另一方面初赛要求运行官方给定的二进制程序, 因此, 这个方案在初赛中并不适用. 综上所述, Linux Subsystem 是一个折中的方案, 既可以实现 Linux 兼容, 又可以避免大量的工作量, 并且可以在初赛中使用.
